# 公司社群平台測試計畫

## 文件目的

這份文件用來說明專案目前的測試範圍、分層方式與執行原則。內容以展示專案品質設計為主，不放內部規劃筆記。

## 專案背景

本系統是一個公司內部社群平台。員工可以發文、留言、查看活動紀錄；若內容命中管理者設定的關鍵字，系統會依風險等級啟動一層到三層審核流程。

## 測試目標

- 驗證主要使用流程可正常操作
- 驗證後端 API 與資料流程正確
- 驗證角色權限限制確實生效
- 驗證關鍵字觸發與分層審核流程符合設計
- 在 CI 中自動執行核心檢查，降低人工遺漏風險

## 測試分層

### 1. Build 驗證

目的：確認前端與多環境設定可成功建置。

主要指令：

- `npm run build`
- `npm run build:sit`
- `npm run build:uat`
- `npm run build:preprod`

### 2. API Smoke Test

目的：用最小但關鍵的 API 測試快速確認後端主流程仍可使用。

主要驗證：

- `/health` 與 `/meta` 正常回應
- `Viewer` 不能建立貼文
- `Member` 可以建立一般貼文
- 命中關鍵字的貼文會進入審核
- `Moderator` 可以管理關鍵字規則

主要檔案：

- `tests/api/api-smoke.test.mjs`

### 3. DB Integrity Test

目的：確認資料表、種子資料、讀寫流程與相容性沒有明顯問題。

主要驗證：

- schema 初始化
- 核心資料表存在
- 種子資料可讀取
- 基本新增與查詢
- 舊版 schema 遷移路徑

主要檔案：

- `tests/db/db-integrity.test.mjs`

### 4. Performance Smoke

目的：用輕量方式檢查 API 是否出現明顯效能退化。

主要驗證：

- `/health`
- `/meta`
- `/review-tasks`
- p95 latency 門檻
- 最低吞吐量
- 無 request error 與非 2xx 回應

主要檔案：

- `scripts/perf-smoke.mjs`

### 5. Web E2E 功能測試

目的：從使用者操作角度驗證前後端整合流程。

blocking suite：

- `tests/e2e/case-detail-attachments-comments.spec.mjs`
- `tests/e2e/cases-search-history.spec.mjs`
- `tests/e2e/comment-review-moderation.spec.mjs`
- `tests/e2e/profile-settings-drilldown.spec.mjs`
- `tests/e2e/role-403.spec.mjs`

主要驗證：

- 登入與導頁
- 貼文詳情與 reopen continuity
- 搜尋與歷程切換
- 多角色審核流程
- 設定與個人頁 drill-down
- API 權限拒絕

### 6. Mobile Smoke Test

目的：確認 mobile 主要導覽與核心流程正常，而不把所有視覺證據都放進 blocking gate。

blocking suite：

- `tests/e2e/mobile-smoke.spec.mjs`

主要驗證：

- mobile tab navigation
- search entry
- thread detail access
- admin 角色下的 settings、rules、admin reachability

### 7. 視覺與展示型證據

目的：保留畫面與角色對照證據，方便展示與人工 review。

主要檔案：

- `tests/e2e/front-end-visual.spec.mjs`
- `tests/e2e/mobile-iphone-visual.spec.mjs`
- `tests/e2e/mobile-pixel-visual.spec.mjs`
- `tests/e2e/role-matrix.spec.mjs`
- `tests/e2e/self-healing-showcase.spec.mjs`
- `tests/mvp-smoke.spec.mjs`

這些測試保留價值很高，但預設不放進最小 blocking gate。

## E2E 設計原則

本專案的 E2E 採混合策略：

- 真實 UI click 與導覽：驗證使用者旅程
- API setup：縮短前置準備時間
- route reopen：穩定重現 thread、review 或 detail continuity

這樣做的原因是：

- 避免 setup 過長造成 flaky
- 把測試時間留給真正重要的行為
- 讓 CI 更穩定

## CI 納管內容

目前 blocking quality gates 包含：

- web build
- API smoke
- DB integrity
- performance smoke
- web E2E
- mobile smoke

另外保留展示型證據：

- visual evidence
- role matrix evidence
- locator resilience showcase

## 目前限制與風險

- 目前已完成 Playwright 行動版截圖；原生 iOS / Android 真機與模擬器截圖依賴 CI 或具備對應 runtime 的本機環境。
- 這台本機缺少 Android SDK / adb，且缺少可用 iOS Simulator platform，因此原生測試目前已完成骨架與 CI 腳本，但未在本機完整跑通。
- 後端目前使用 SQLite，正式環境仍建議改為受管 PostgreSQL。
- 尚未納入壓力測試、資安掃描與滲透測試。
- 尚未納入真實 APNs / FCM 推播情境。

## 建議部署前補強項目

- 加入 API contract test
- 加入更多 mobile 實機驗證
- 加入效能測試與安全性檢查
- 在 pre-PROD 導入與正式環境一致的資料庫類型

## 結論

目前本專案已具備桌面 Web、iOS、Android 三端的基礎自動化測試骨架，能覆蓋前端主流程、後端 API、角色權限、關鍵字規則、多層審核與 mobile tab 導覽。若以部署前品質控管來看，已達到可持續驗證與可持續回歸的基本要求；後續若要進入正式上線，建議再補上資料庫、實機與資安層面的測試。
