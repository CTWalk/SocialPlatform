# 公司社群平台測試計畫

## 文件目的

本文件用於說明本專案在部署前的測試策略、測試工具、測試範圍與執行方式。內容同時涵蓋桌面 Web、iOS、Android、角色權限、流程驗證、視覺證據與 CI/CD 自動化，作為專案報告的一部分。

## 專案背景

本系統是一個公司內部社群平台。一般員工可在公開區域發文；若貼文內容命中管理者設定的關鍵字，系統會依風險等級觸發一層、兩層或三層審核流程，通過後才可發布。

## 測試目標

- 驗證前端主要流程可正常操作。
- 驗證後端 API 與資料流程正確。
- 驗證多角色權限控制確實生效。
- 驗證關鍵字觸發與多層審核流程符合設計。
- 驗證 SIT、UAT、pre-PROD 環境都可成功建置。
- 在 CI/CD 中自動執行核心測試，降低人工遺漏風險。

## 分支與測試歸屬

- `develop` 是開發主線，同時代表 SIT integration truth
- `release/uat` 對應 UAT promotion
- `release/preprod` 對應 pre-PROD promotion
- `main` 對應 production
- `showcase` 只作為 demo / presentation surface，不作為工程真相來源

測試策略與品質門檻以 Development family 為準，再將通過驗證的結果同步到 `showcase`。

## 測試範圍

### 前端測試範圍

- 登入流程
- Dashboard 顯示
- Comment Board 發文與檢視
- Post Detail 留言與歷程
- Review Queue
- Keyword Rules 管理
- Administration 使用者管理
- RWD / mobile 模擬畫面
- iOS 原生 shell 與 XCTest / XCUI Test
- Android 原生 shell 與 Espresso / Instrumentation

### 後端測試範圍

- 健康檢查與基本 metadata
- 登入 / 登出 API
- 貼文建立、發布、審核、拒絕
- 關鍵字規則建立與切換
- 使用者建立與查詢
- 角色權限限制
- SQLite 資料寫入與讀取

## 測試策略

### 1. Build 驗證

目的：確認 Angular 前端、Capacitor 原生殼與多環境建置在目前程式碼狀態下可成功編譯，避免進入測試前就已存在結構性錯誤。

使用方式：

- 指令：`npm run build`
- 環境建置指令：
- `npm run build:sit`
- `npm run build:uat`
- `npm run build:preprod`
- `npm run ios:build:prod`
- `npm run android:build:prod`

### 2. API Smoke Test

目的：用最小但關鍵的 API 測試，快速確認後端主流程仍可使用。

使用工具：

- Node.js 內建 `node:test`
- 測試檔：`tests/api/api-smoke.test.mjs`

主要驗證內容：

- `/health` 與 `/meta` 可回應
- `Viewer` 無法建立貼文
- `Member` 可建立一般貼文
- 命中關鍵字的貼文會進入分層審核
- `Moderator` 可管理關鍵字規則

執行指令：

- `npm run test:api`

### 3. E2E 功能測試

目的：從使用者操作角度驗證前後端整合流程。

使用工具：

- Playwright
- 測試資料夾：`tests/e2e/`

主要驗證內容：

- 登入與導頁
- 發文
- 搜尋與篩選
- 開啟貼文詳情
- 留言與歷程
- 權限限制
- 多角色檢視差異
- mobile 底部導覽依角色切換

執行指令：

- `npm run test:e2e`
- `npm run test:smoke`

### 4. 視覺證據測試

目的：保存可視化證據，讓報告、驗收與回歸比對更容易。

使用工具：

- Playwright screenshot
- XCTest screenshot attachment
- Android instrumentation / emulator artifact

主要證據：

- 桌面版主要流程：`tests/e2e/front-end-visual.spec.mjs`
- 行動版主要流程：
  - `tests/e2e/mobile-iphone-visual.spec.mjs`
  - `tests/e2e/mobile-pixel-visual.spec.mjs`
- iOS 原生測試：
  - `ios/App/AppTests/`
  - `ios/App/AppUITests/`
- Android 原生測試：
  - `android/app/src/androidTest/java/com/company/socialplatform/`

輸出位置：

- 桌面版：`tests/screenshots/`
- 行動版 Playwright：`tests/screenshots/mobile/playwright/`
- iOS Simulator：`tests/screenshots/mobile/ios-simulator/`
- Android Emulator：`tests/screenshots/mobile/android-emulator/`

執行指令：

- `npm run test:visual`
- `npm run test:mobile`
- `npm run ios:test:sim`
- `npm run android:test:emulator`

### 5. 權限與角色驗證

目的：證明系統不是只有畫面顯示角色，而是真的依角色限制功能與資料操作。

使用工具：

- Playwright
- API header 驗證

主要測試檔：

- `tests/e2e/role-matrix.spec.mjs`
- `tests/e2e/role-403.spec.mjs`
- `tests/role-permission-matrix.md`

驗證重點：

- Administrator 可管理使用者
- Moderator 可管理關鍵字規則
- Member 可發文
- Reviewer / Approver / Moderator 可執行審核
- Viewer 為唯讀，不可建立貼文或使用者

### 6. 原生行動測試

目的：驗證 iOS / Android 原生 App shell 與行動版 UI 導覽可正常載入，並能覆蓋主要旅程。

使用工具：

- iOS：`XCTest`、`XCUI Test`
- Android：`Espresso`、`Instrumentation`

主要驗證內容：

- App launch
- session restore
- mobile 底部導覽
- Comment Board / Search / Review / Rules / Profile 切換
- 原生殼成功載入 WebView

執行指令：

- `npm run ios:test:sim`
- `npm run android:test:emulator`

### 7. Device Farm / Test Lab

目的：將原生測試產物送到裝置雲，驗證真機或雲端裝置環境中的可執行性。

使用工具：

- AWS Device Farm
- Firebase Test Lab

主要驗證內容：

- iOS XCUI package 上傳
- Android instrumentation APK 上傳
- 觸發雲端裝置測試

執行指令：

- `npm run ios:test:devicefarm`
- `npm run android:test:devicefarm`
- `npm run android:test:testlab`

## 測試環境規劃

### SIT

目的：系統整合測試，確認前後端整合與主要流程正常。

驗證內容：

- Build
- API smoke
- E2E smoke
- 主要角色權限
- iOS / Android 基礎建置

### UAT

目的：提供業務或使用者代表驗收主要功能與操作流程。

驗證內容：

- 關鍵字觸發邏輯
- 分層審核流程
- 角色可見性
- 報表與管理頁基礎流程
- mobile 底部導覽與角色切換

### pre-PROD

目的：以接近正式環境的設定再次驗證部署前品質。

驗證內容：

- 環境建置
- API smoke
- 完整 E2E
- iOS / Android 原生 build
- Device Farm / Test Lab pipeline

## 目前限制與風險

- 目前已完成 Playwright 行動版截圖；原生 iOS / Android 真機與模擬器截圖依賴 CI 或具備對應 runtime 的本機環境。
- 這台本機缺少 Android SDK / adb，且缺少可用 iOS Simulator platform，因此原生測試目前已完成骨架與 CI 腳本，但未在本機完整跑通。
- 後端目前使用 SQLite，正式環境仍建議改為受管 PostgreSQL。
- 尚未納入壓力測試、資安掃描與滲透測試。
- 尚未納入真實 APNs / FCM 推播情境。

## 建議部署前補強項目

- 加入 API contract test
- 加入更多 mobile 實機驗證
- 加入效能測試與安全性檢查
- 在 pre-PROD 導入與正式環境一致的資料庫類型

## 結論

目前本專案已具備桌面 Web、iOS、Android 三端的基礎自動化測試骨架，能覆蓋前端主流程、後端 API、角色權限、關鍵字規則、多層審核與 mobile tab 導覽。若以部署前品質控管來看，已達到可持續驗證與可持續回歸的基本要求；後續若要進入正式上線，建議再補上資料庫、實機與資安層面的測試。
